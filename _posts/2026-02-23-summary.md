### [Attention Media ≠ Social Networks](https://susam.net/attention-media-vs-social-networks.html) ⭐️ 7.0/10

On January 20, 2026, Susam Pal published a conceptual analysis distinguishing 'attention media' platforms, which are designed to maximize user engagement for advertising revenue, from genuine 'social networks,' which are designed to facilitate connections between people. This distinction matters because it reframes how we evaluate digital platforms, revealing that many services labeled as social networks are actually optimized for extracting user attention rather than fostering meaningful social interaction, which has profound implications for user well-being, privacy, and the design of future online communities. A key technical detail is that the design and terminology of a platform reflect its priorities; for instance, the shift from calling services 'social networks' to 'social media' often signals a move from friend-centric connections to broadcast-style content and ads.

*Source: hackernews/susam*

**Background**: The 'attention economy' is a concept describing how digital platforms, particularly those funded by advertising, are incentivized to maximize the time and attention users spend on them. Traditional 'social networks' were originally designed as digital spaces to map and maintain real-world social connections between individuals. In contrast, many modern platforms prioritize algorithmic feeds of engaging content from both friends and publishers to capture user attention, which is then monetized.

**Discussion**: Community comments reflect agreement with the core distinction, with users noting the shift in terminology from 'social networks' to 'social media' signaled a move from friends to fame and ads. Some users point out that platforms like Facebook contain a 'genuine social network' feature (like the 'Feeds' tab) buried within the attention-media interface, while others debate alternatives, noting that even decentralized platforms like Mastodon can inherit bad habits from mainstream social media.

### [I built Timeframe, our family e-paper dashboard](https://hawksley.org/2026/02/17/timeframe.html) ⭐️ 6.0/10

A developer named Hawksley built a custom family dashboard called 'Timeframe' using e-paper displays to share calendar events, weather, and other information without traditional backlit screens, as documented in a blog post from February 17, 2026. This project demonstrates a practical, low-power alternative to conventional screens for ambient home information displays, addressing the desire for shared family information without the distractions and energy consumption of phones or tablets. The project's primary display is a large e-paper unit costing approximately $2000, which is a significant barrier to widespread adoption, and the system was built as a custom DIY solution rather than using off-the-shelf consumer products.

*Source: hackernews/saeedesmaili*

**Background**: E-paper, or electronic paper, is a display technology that mimics the appearance of ordinary ink on paper and is known for being extremely energy-efficient, as it only consumes power when the image changes. DIY home automation often involves creating custom dashboards to centralize control and information, with platforms like Home Assistant being popular choices. Information display hardware projects frequently use components like e-ink displays, microcontrollers, and custom software to show data like weather, calendars, or system status.

**Discussion**: The community expressed admiration for the project's concept but focused heavily on the high cost of large e-paper displays, attributing it to patent issues that stifle competition and innovation. Several commenters proposed alternative, lower-cost solutions like reverse-engineering old retail displays or even using a simple pane of glass as a dry-erase board, highlighting a desire for the benefits of e-paper without the prohibitive expense.

**Tags**: `#e-paper`, `#home-automation`, `#DIY`, `#information-display`, `#hardware`

### [How I think about Codex](https://simonwillison.net/2026/Feb/22/how-i-think-about-codex/#atom-everything) ⭐️ 6.0/10

An OpenAI developer, Gabriel Chua, clarified the confusing terminology around 'Codex,' defining it as OpenAI's software engineering agent composed of three parts: Model, Harness, and Surfaces. He also confirmed that Codex models are uniquely trained in the presence of the specific harness, integrating tool use and execution loops directly into their learning. This clarification matters because it resolves widespread confusion in the developer community about what 'Codex' refers to, distinguishing between the core agent system and its various interfaces. Understanding this architecture is key for developers looking to build with or integrate OpenAI's agentic coding tools effectively. A key technical detail is that the 'Harness'—the collection of instructions and tools that forms part of the agent—is open source and available in the `openai/codex` GitHub repository. Furthermore, the bidirectional JSON-RPC protocol used in the related App Server architecture enables a continuous, real-time coding experience with features like streaming progress and tool use.

*Source: rss/Simon Willison*

**Background**: OpenAI Codex is a system designed to act as a software engineering agent, capable of assisting with or autonomously performing coding tasks. An 'agent' in this context typically refers to an AI system that combines a large language model (LLM) with specific instructions and tools, wrapped in a runtime that can execute tasks. The term 'Codex' has been used ambiguously, sometimes referring to a model family (like GPT-4 Codex) and sometimes to the broader agent system, which has caused confusion.
